## What this is

A single-process **Bun + Elysia + TypeScript** service that exposes a self-hosted
[BlueBubbles](https://bluebubbles.app) iMessage server as a **custom channel**
inside [Missive](https://missiveapp.com) — bidirectionally, without echo loops or
duplicates. See `README.md` for the full product description, setup, and the
Missive channel-creation walkthrough.

## Golden rules (non-negotiable)

1. **Update the docs** with every change (see the directive above).
2. **The 100% coverage gate is not optional.** `bun test --coverage` must pass
   with 100% line + function coverage (enforced by `bunfig.toml`). New code needs
   new tests; don't lower `coverageThreshold` or widen `coveragePathIgnorePatterns`
   to get green.
3. **Respect the [8 correctness invariants](#the-8-correctness-invariants).** They
   are load-bearing and enforced by tests. If you touch dedup, the outbox, echo
   suppression, threading, HMAC, or the send paths, re-read them first.
4. **Keep the layering.** Pure transforms in `domain/*`; all network/DB I/O in
   `clients/*` and `queue/*`; routes only verify → dedup+enqueue → ack. Don't do
   outbound I/O on the request path.
5. **Conventional Commits**, and commit/PR messages contain **no AI-assistant
   attribution** (no "Generated by", no `Co-Authored-By` bot lines) — match the
   existing history.
6. **Everything lands via a green PR.** CI (typecheck + lint + 100% coverage)
   runs on every PR and posts a sticky coverage comment.

## Commands

```bash
bun install                 # install deps (CI uses --frozen-lockfile)
bun run dev                 # watch-mode server (src/index.ts)
bun run start               # run once
bun run typecheck           # tsc --noEmit (strict)
bun run lint                # biome check .   (lint + format verification)
bun run format              # biome format --write .
bun test                    # unit + integration + e2e (coverage on via bunfig)
bun run test:cov            # explicit coverage run (same gate)
```

**The full local gate before pushing** (mirrors CI):

```bash
bun install --frozen-lockfile && bun run typecheck && bun run lint && bun test --coverage
```

**Docker** (see `README.md#run-with-docker`):

```bash
docker compose up -d --build   # build + run the bridge (state on the bridge-data volume)
docker compose logs -f         # watch the boot sequence
docker compose down            # stop (keeps the volume; `-v` also drops state)
```

`Dockerfile` (single stage: no build step — Bun runs TS directly, `bun:sqlite`
is built in; installs `--production` deps; runs as non-root `bun`), plus a
bridge-only `docker-compose.yml` (persistent `bridge-data` volume, `.env` via
`env_file`, `BB_URL` defaults to `host.docker.internal` but `.env` overrides).
The tunnel for `PUBLIC_URL` stays external. `.dockerignore` keeps `.env`, state,
and tests out of the image.

## Architecture

```
src/
  index.ts            compose Elysia; onStart: ping → detect caps → self-register
                      BB webhook → start worker; bootstrap guarded by import.meta.main
  config.ts           the ONLY reader of process.env; validates + freezes; fail-fast
  types.ts            shared type contract (runtime-free; import type)
  util.ts             verifyHmac, msToUnix, backoffMs, canonicalHash, bbDedupKey
  logger.ts           leveled JSON-line logs; redacts secrets
  db.ts               bun:sqlite singleton: schema, PRAGMAs, prepared stmts, helpers
  routes/
    missive-webhook.ts  parse:'none' → raw-body HMAC → dedupAndEnqueue → 200
    bb-webhook.ts       token guard → per-type dedup key → dedupAndEnqueue → 200
    health.ts           GET /health, GET /  (caps, lastProbeAt, outboxDepth, ready)
  clients/
    bluebubbles.ts      transport only: sendText/Attachment, chatNew, queries,
                        downloadAttachment, serverInfo, webhook mgmt, PA: react/edit/…
    missive.ts          callMissive (Bearer, 429 Retry-After), postInboundMessage, posts
  domain/               PURE transforms + small caches (the main unit-test surface)
    inbound.ts          BB event → Missive "messages" body/bodies (+ tapback/edit/group render)
    outbound.ts         Missive webhook → BB send plan (resolve chatGuid, reply vs new)
    identity.ts         address → contact name (cached)
    capability.ts       Private-API detection (private_api && helper_connected) + re-probe
  queue/
    ratelimiter.ts      Missive token bucket (≤5 concurrent, ~1 req/s, burst 5)
    outbox.ts           durable worker: claimDueJobs (per-chat barrier) → dispatch → retry
test/                   unit + integration (mock Missive & BB via Bun.serve) + e2e smoke
```

**Two flows** (details in `README.md`):

- **Missive → BlueBubbles:** signed webhook → HMAC verify → enqueue → worker resolves
  chatGuid (reply vs new) → send text/attachment via the BlueBubbles client.
- **BlueBubbles → Missive:** unsigned webhook (token-guarded) → dedup → enqueue → worker
  maps the event → downloads attachments → `POST /v1/messages` with `references` threading.

Both directions are ack-fast; all side effects happen in the durable **outbox** worker.

## Data model (`bun:sqlite`, 6 tables)

`chat_map` (chatGuid ↔ Missive conversation/reference), `handle_map` (address → name),
`message` (guid → chatGuid + text; needed because `updated-message` has no `chats[]`),
`sent_map` (outbound idempotency + echo correlation), `seen_events` (dedup),
`outbox` (durable job queue; `chat_guid` column powers the per-chat barrier).
`db.ts` owns the schema, all prepared statements, and the transactional helpers.

## The 8 correctness invariants

These are **requirements**, encoded in the foundation and enforced by tests. Full
prose lives in `README.md#the-8-correctness-invariants`; the short list:

1. **Atomic dedup+enqueue** — `Db.dedupAndEnqueue` runs `firstSeen` + outbox insert in one transaction.
2. **Type-correct dedup keys** — `util.bbDedupKey`; never `…:undefined:undefined`; ephemeral events → `null`.
3. **No double-send on new conversations** — one send path (`chat/new` *with* body **xor** `message/text`); record its `tempGuid`.
4. **Don't assume `chat/new` is idempotent** — resolve an existing chat first.
5. **Echo suppression by consume-on-match** — `Db.consumeEcho` (atomic); `tempGuid` is NOT in the echo webhook, so it can't be the correlator.
6. **Bind conversation id when known** — always send `references`; add `conversation` once `chat_map` has it.
7. **Per-POST unique `external_id`** — a split BB message → N outbox jobs, each with its own id + delivery ledger.
8. **Outbound retry reuses the stored `tempGuid`** — BlueBubbles `sendCache` dedups the resend.

## Conventions

- **TypeScript strict**, ESM. No `any` in exported signatures — use types from `src/types.ts`.
- **Pure `domain/*`**: no network/DB there; return plans, let the worker perform I/O. This is what keeps them unit-testable.
- **Testability**: export units; guard the real bootstrap behind `if (import.meta.main)`; inject the clock/fetch where needed so 100% coverage is reachable without hitting the network.
- **Secrets**: never log `BB_PASSWORD`, `MISSIVE_TOKEN`, `MISSIVE_HMAC_SECRET`, `BB_HOOK_TOKEN`. `logger.ts` redacts; keep it that way.
- **Formatting/lint**: Biome. Run `bun run format` before committing; `bun run lint` must be clean.
- **Config**: `src/config.ts` is the single source of truth for env. Add new vars there, document them in `README.md` **and** `.env.example`.

## Testing & the coverage gate

- `bunfig.toml` sets `coverage = true` and `coverageThreshold = 1.0`, so `bun test`
  itself exits non-zero below 100%. `test/setup.ts` is preloaded to give `config.ts`
  a valid env at import time. Only `test/**` and the runtime-free `src/types.ts` are
  excluded from measurement.
- Mock external HTTP with a local `Bun.serve` (see `test/support/mocks.ts`) or an
  injected `fetch` — never call the real network in tests.
- When you add a branch, add a test that would **fail if the branch regressed**
  (meaningful coverage, not line-execution for its own sake). Cover new branches of
  the 8 invariants explicitly.

## CI/CD & Dependabot

- **`.github/workflows/ci.yml`** — one `quality` job on PR / push-to-main / manual:
  `bun install --frozen-lockfile` → typecheck → lint → `bun test --coverage`, then
  upserts a sticky coverage comment on PRs. Least-privilege permissions; all dynamic
  values pass through `env:` (no script injection).
- **`.github/dependabot.yml`** uses the **`bun`** ecosystem (not `npm`). This matters:
  the `npm` ecosystem bumps `package.json` but leaves `bun.lock` stale, which breaks
  `--frozen-lockfile`. Keep it `bun`.
- If you change deps, commit the updated **`bun.lock`** alongside `package.json`
  (run `bun install`, not a manual edit).

## Gotchas (things that will bite you)

- **`bun install --frozen-lockfile` failing on a Dependabot PR** → the lockfile is
  out of sync; usually the wrong Dependabot ecosystem. Ecosystem is `bun` — keep it.
- **`updated-message` webhooks have no `chats[]`** → resolve `chatGuid` from the
  `message` table (populated by the earlier `new-message`). Don't read `data.chats`.
- **Echo loop** → messages the bridge sends bounce back as `new-message` with
  `isFromMe:true`; they must be suppressed via `consumeEcho`, not re-posted.
- **HMAC must be computed on the raw request bytes** (`parse:'none'` +
  `arrayBuffer()`), never on re-serialized JSON — key order/whitespace differ.
- **Missive rate limits** (5 concurrent, ~1 req/s sustained). Route Missive calls
  through the token-bucket limiter and honor `429 Retry-After`.
- **`POST /v1/messages` may return an empty 201** — never depend on its response
  body for ids; correlation is client-side.
